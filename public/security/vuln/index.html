Gestión de Vulnerabilidades en Go

Volver a Seguridad en Go

Descripción general

Go ayuda a los desarrolladores a detectar, evaluar y resolver errores o debilidades que corren el riesgo de ser explotados por atacantes. Detrás de escena, el equipo de Go ejecuta un proceso para seleccionar informes sobre vulnerabilidades, que se almacenan en la base de datos de vulnerabilidades de Go. Varias bibliotecas y herramientas pueden leer y analizar esos informes para comprender cómo pueden verse afectados proyectos de usuarios específicos. Esta funcionalidad está integrada en el sitio de descubrimiento de paquetes Go y en una nueva herramienta CLI, govulncheck.

Este proyecto es un trabajo en progreso y en desarrollo activo. ¡Agradecemos tus comentarios para ayudarnos a mejorar!

NOTA: Para informar una vulnerabilidad en el proyecto Go, consulta la Política de seguridad de Go.

Arquitectura

La gestión de vulnerabilidades en Go consta de las siguientes piezas de alto nivel:

  1. Un data pipeline recopila información sobre vulnerabilidades de varias fuentes, incluido el Base de datos nacional de vulnerabilidades (NVD), la Base de datos de asesoramiento de GitHub y directamente de los mantenedores del paquete Go ⬀.
  2. Una base de datos de vulnerabilidad se llena con informes que utilizan información del data pipeline. Todos los informes de la base de datos son revisados y seleccionados por el equipo de Go Security. Los informes tienen el formato formato de vulnerabilidad de código abierto (OSV) y se puede acceder a ellos a través de la API.
  3. Integraciones con pkg.go.dev y govulncheck para permitir a los desarrolladores encontrar vulnerabilidades en sus proyectos. El comando govulncheck analiza tu código base y solo muestra vulnerabilidades que realmente te afectan, en función de qué funciones en tu código llaman transitivamente a funciones vulnerables. Govulncheck proporciona una manera confiable y silenciosa de encontrar vulnerabilidades conocidas en tus proyectos.

Recursos

Base de datos de vulnerabilidades de Go

La base de datos de vulnerabilidades de Go contiene información de muchos fuentes existentes además de los informes directos de los mantenedores de paquetes de Go al equipo de seguridad de Go. Cada entrada en la base de datos se revisa para garantizar que la descripción de la vulnerabilidad, la información del paquete y del símbolo, y los detalles de la versión sean precisos.

Consulta go.dev/security/vuln/database para obtener más información sobre la base de datos de vulnerabilidades de Go, y pkg.go.dev/vuln para ver las vulnerabilidades en la base de datos de tu navegador.

Animamos a los mantenedores de paquetes a contribuir con información sobre vulnerabilidades públicas en sus propios proyectos y envíenos sugerencias sobre cómo reducir la fricción.

Detección de vulnerabilidades para Go

La detección de vulnerabilidades de Go tiene como objetivo proporcionar una forma confiable y silenciosa para que los usuarios de Go conozcan las vulnerabilidades conocidas que pueden afectar sus proyectos. La verificación de vulnerabilidades está integrada en las herramientas y servicios de Go, incluida una nueva herramienta de línea de comandos, govulncheck, el sitio de descubrimiento de paquetes Go, editores importantes como VS Code con la extensión Go.

Para comenzar a usar govulncheck, ejecuta lo siguiente desde tu proyecto:

$ go install golang.org/x/vuln/cmd/govulncheck@latest
$ govulncheck ./...

Para habilitar la detección de vulnerabilidades en tu editor, consulta las instrucciones en la página integración del editor.

Go CNA

El equipo de seguridad de Go es una Numeración CVE Autoridad. Consulta go.dev/security/vuln/cna para obtener más información.

FAQs

¿Cómo reporto una vulnerabilidad en el proyecto Go?

Reportar todos los errores de seguridad en el proyecto Go por correo electrónico a security@golang.org. Lee la Política de seguridad de Go para obtener más información sobre nuestros procesos.

¿Cómo agrego una vulnerabilidad pública a la base de datos de vulnerabilidades de Go?

Para solicitar la adición de una vulnerabilidad pública a la base de datos de vulnerabilidades de Go, completa este formulario ⬀.

Una vulnerabilidad se considera pública si ya se ha divulgado públicamente o si existe en un paquete que mantienes (y estás listo para divulgarlo). El formulario es solo para vulnerabilidades públicas en paquetes de Go importables que no son mantenidos por el equipo de Go (cualquier cosa fuera de la biblioteca estándar de Go, la cadena de herramientas de Go y los módulos de golang.org).

El formulario también se puede utilizar para solicitar una nueva CVE ID. Lee más aquí sobre la autoridad de numeración Go CVE.

¿Cómo sugiero una edición de una vulnerabilidad?

Para sugerir una edición de un informe existente en la base de datos de vulnerabilidades de Go, completa el formulario aquí ⬀.

¿Cómo informo un problema o doy comentarios sobre govulncheck?

Envía tu problema o comentario en el rastreador de issues de Go ⬀.

Encontré esta vulnerabilidad en otra base de datos. ¿Por qué no está en la base de datos de vulnerabilidades de Go?

Los informes pueden excluirse de la base de datos de vulnerabilidades de Go por varias razones, incluida la vulnerabilidad relevante que no está presente en un paquete de Go, que la vulnerabilidad está en un comando instalable en lugar de un paquete importable, o la vulnerabilidad está subsumida por otra vulnerabilidad que ya está presente en la base de datos. Puede obtener más información sobre los motivos para excluir informes del equipo de Go Security aquí. Si crees que un informe se excluyó incorrectamente de vuln.go.dev, háznoslo saber ⬀.

¿Por qué la base de datos de vulnerabilidades de Go no usa etiquetas de gravedad?

La mayoría de los formatos de informes de vulnerabilidades utilizan etiquetas de gravedad como "BAJO", "MEDIO" y "CRÍTICO" para indicar el impacto de diferentes vulnerabilidades y ayudar a los desarrolladores a priorizar los problemas de seguridad. Sin embargo, por varias razones, Go evita el uso de este tipo de etiquetas.

El impacto de una vulnerabilidad rara vez es universal, lo que significa que los indicadores de gravedad a menudo pueden ser engañosos. Por ejemplo, una falla en un analizador puede ser un problema de gravedad crítico si se usa para analizar la entrada proporcionada por el usuario y se puede aprovechar en un ataque DoS, pero si el analizador se usa para analizar archivos de configuración locales, incluso llamar a la gravedad "bajo” podría ser una exageración.

La severidad del etiquetado también es necesariamente subjetiva. Esto es cierto incluso para el programa CVE, que propone una fórmula para descomponer aspectos relevantes de una vulnerabilidad, como el vector de ataque, la complejidad y la explotabilidad. Todo esto, sin embargo, requiere una evaluación subjetiva.

Creemos que las buenas descripciones de las vulnerabilidades son más útiles que los indicadores de gravedad. Una buena descripción puede desglosar qué es un problema, cómo se puede desencadenar y qué deben considerar los consumidores al determinar el impacto en su propio software.
