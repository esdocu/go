Base de datos de vulnerabilidades de Go

Volver a Gestión de vulnerabilidades de Go

Descripción general

La base de datos de vulnerabilidades de Go (https://vuln.go.dev) proporciona información sobre la vulnerabilidad de Go en Vulnerabilidad de código abierto (OSV) esquema.

También puedes buscar vulnerabilidades en la base de datos en pkg.go.dev/vuln.

No confíes en el contenido del repositorio Git x/vulndb. Los archivos YAML en ese repositorio se mantienen usando un formato interno que puede cambiar sin previo aviso.

Contribuyendo

Nos encantaría que todos los mantenedores de paquetes Go contribuyan ⬀ con información sobre vulnerabilidades públicas en sus propios proyectos y actualicen ⬀ la información existente sobre las vulnerabilidades en sus paquetes Go.

Nuestro objetivo es que la presentación de informes sea un proceso de baja fricción, así que no dudes en envíanos tus sugerencias ⬀.

Por favor, no utilices los formularios anteriores para informar una vulnerabilidad en la biblioteca estándar o los subrepositorios de Go. En su lugar, sigue el proceso en go.dev/security/policy para conocer las vulnerabilidades del proyecto Go.

API

La base de datos canónica de vulnerabilidades de Go, https://vuln.go.dev, es un servidor HTTP que puede responder a solicitudes GET para los endpoints especificados a continuación.

Los endpoints no tienen parámetros de consulta y no se requieren encabezados específicos. Debido a esto, incluso un sitio que sirve desde un sistema de archivos fijo (incluida una URL file:// ) puede implementar esta API.

Cada endpoint devuelve una respuesta codificada en JSON, ya sea sin comprimir (si se solicita como .json) o en formato comprimido (si se solicita como .json.gz).

Los endpoints son:

  • /index/db.json[.gz]

    Devuelve metadatos sobre la base de datos:

    {
  // The latest time the database should be considered
  // to have been modified, as an RFC3339-formatted UTC
  // timestamp ending in "Z".
  "modified": string
}

    Ten en cuenta que la hora modificada no debe compararse con la hora del reloj de pared, por ejemplo, para fines de invalidación de caché, ya que puede haber un retraso en la realización de modificaciones de la base de datos.

    Ver /index/db.json para ver un ejemplo en vivo.

  • /index/modules.json[.gz]

    Devuelve una lista que contiene metadatos sobre cada módulo en la base de datos:

    [ {
  // The module path.
  "path": string,
  // The vulnerabilities that affect this module.
  "vulns":
    [ {
      // The vulnerability ID.
      "id": string,
      // The latest time the vulnerability should be considered
      // to have been modified, as an RFC3339-formatted UTC
      // timestamp ending in "Z".
      "modified": string,
      // (Optional) The module version (in SemVer 2.0.0 format)
      // that contains the latest fix for the vulnerability.
      // If unknown or unavailable, this should be omitted.
      "fixed": string,
    } ]
} ]

    Ver /index/modules.json para ver un ejemplo en vivo.

  • /index/vulns.json[.gz]

    Devuelve una lista que contiene metadatos sobre cada vulnerabilidad en la base de datos:

     [ {
     // The vulnerability ID.
     "id": string,
     // The latest time the vulnerability should be considered
     // to have been modified, as an RFC3339-formatted UTC
     // timestamp ending in "Z".
     "modified": string,
     // A list of IDs of the same vulnerability in other databases.
     "aliases": [ string ]
 } ]

    Ver /index/vulns.json para ver un ejemplo en vivo.

  • /ID/$id.json[.gz]

    Devuelve el informe individual de la vulnerabilidad con ID $id, en formato OSV (descrito a continuación en Esquema).

    Ver /ID/GO-2022-0191.json para ver un ejemplo en vivo.

Uso en govulncheck

De forma predeterminada, govulncheck usa la base de datos canónica de vulnerabilidades de Go en vuln.go.dev.

El comando se puede configurar para contactar una base de datos de vulnerabilidad diferente usando el indicador -db, que acepta una URL de base de datos de vulnerabilidad con protocolo http://, https:// o file://.

Para funcionar correctamente con govulncheck, la base de datos de vulnerabilidades especificada debe implementar la API descrita anteriormente. El comando govulncheck utiliza puntos finales comprimidos “.json.gz” cuando lee desde una fuente http(s), y los puntos finales “.json” cuando lee desde una fuente de archivo.

Legacy API

La base de datos canónica contiene algunos endpoints adicionales que son parte de una Legacy API. Planeamos eliminar pronto la compatibilidad con estos endpoints. Si utilizas la Legacy API y necesitas tiempo adicional para migrar, háganoslo saber ⬀.

Esquema

Los informes utilizan el esquema de Vulnerabilidad de código abierto (OSV). La base de datos de vulnerabilidades de Go asigna los siguientes significados a los campos:

id

El campo id es un identificador único para la entrada de vulnerabilidad. Es una cadena con el formato GO-<YEAR>-<ENTRYID>.

affected

Los campos afectados es una array JSON que contiene objetos que describen las versiones del módulo que contienen la vulnerabilidad.

affected[].package

El affected[].package es un objeto JSON que identifica el módulo afectado. El objeto tiene dos campos obligatorios:

  • ecosystem: esto siempre será “Go”
  • name: esta es la ruta del módulo Go
    • Los paquetes importables en la biblioteca estándar tendrán el nombre stdlib.
    • El comando go tendrá el nombre toolchain.

affected[].ecosystem_specific

El affected[].ecosystem_specific es un objeto JSON con información adicional sobre la vulnerabilidad, que utilizan las herramientas de detección de vulnerabilidades de Go.

Por ahora, el ecosistema específico siempre será un objeto con un solo campo, imports.

affected[].ecosystem_specific.imports

El campo affected[].ecosystem_specific.imports es una array JSON que contiene los paquetes y símbolos afectados por la vulnerabilidad. Cada objeto del array tendrá estos dos campos:

  • path: una cadena con la ruta de importación del paquete que contiene la vulnerabilidad
  • symbols: un array de cadenas con los nombres de los símbolos (función o método) que contiene la vulnerabilidad
  • goos: un array de cadenas con el sistema operativo de ejecución donde aparecen los símbolos, si se conocen
  • goarch: un array de cadenas con la arquitectura donde aparecen los símbolos, si se conoce

database_specific.url

El campo database_specific.url es una cadena que representa la URL completa del informe de vulnerabilidad de Go, por ejemplo, “https://pkg.go.dev/vuln/GO-2023-1621".

Para obtener información sobre otros campos del esquema, consulta la Especificación OSV.

Informes excluidos

Los informes en la base de datos de vulnerabilidades de Go se recopilan de diferentes fuentes y son seleccionados por el equipo de seguridad de Go. Es posible que nos encontremos con un aviso de vulnerabilidad (por ejemplo, un CVE o GHSA) y decidamos excluirlo por diversas razones. En estos casos, se creará un informe mínimo en el repositorio x/vulndb, en x/vulndb/data/excluded.

Los informes pueden excluirse por estos motivos:

  • NOT_GO_CODE: la vulnerabilidad no está en un paquete Go, pero otra fuente la marcó como un aviso de seguridad para el ecosistema Go. Esta vulnerabilidad no puede afectar a ningún paquete Go. (Por ejemplo, una vulnerabilidad en una biblioteca de C++).
  • NOT_IMPORTABLE: La vulnerabilidad ocurre en el paquete main, un paquete internal/ solo importado por paquete main, o alguna otra ubicación que nunca pueda ser importada por otro módulo.
  • EFFECTIVELY_PRIVATE: Si bien la vulnerabilidad ocurre en un paquete Go que puede ser importado por otro módulo, el paquete no está diseñado para uso externo y no es probable que nunca sea importado fuera del módulo en el que está definido.
  • DEPENDENT_VULNERABILITY: Esta vulnerabilidad es un subconjunto de otra vulnerabilidad en la base de datos. Por ejemplo, si el paquete A contiene una vulnerabilidad, el paquete B depende del paquete A y hay CVE ID separados para los paquetes A y B, podríamos marcar el informe de B como una vulnerabilidad dependiente completamente reemplazada por el informe de A.
  • NOT_A_VULNERABILITY: si bien se ha asignado un CVE ID o GHSA, no existe ninguna vulnerabilidad conocida asociada a él.

Por el momento, los informes excluidos no se entregan a través de la API vuln.go.dev. Sin embargo, si tienes un caso de uso específico y sería útil tener acceso a esta información a través de la API, háznoslo saber ⬀.
