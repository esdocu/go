Política de Seguridad de Go

Descripción general

Este documento explica el proceso del equipo de Go Security para manejar los problemas reportados y qué esperar a cambio.

Reportar un error de seguridad

Todos los errores de seguridad en la distribución Go deben informarse por correo electrónico a security@golang.org. Este correo se entrega al equipo de Go Security.

Para asegurarte de que tu informe no se marque como spam, incluye la palabra "vulnerabilidad" en cualquier parte de tu correo electrónico. Utiliza una línea de asunto descriptiva para el correo electrónico de tu informe.

Tu correo electrónico será reconocido dentro de los 7 días y se te mantendrá actualizado con el progreso hasta la resolución. Tu problema se solucionará o se hará público dentro de los 90 días.

Si no has recibido una respuesta a tu correo electrónico dentro de 7 días, comunícate nuevamente con el equipo de seguridad de Go en seguridad@golang.org. Asegúrate de que la palabra vulnerabilidad esté en tu correo electrónico.

Si después de 3 días más aún no has recibido un acuse de recibo de tu reporte, es posible que tu correo electrónico haya sido marcado como spam. En ese caso, presenta un issue aquí. Selecciona “Quiero informar de un error técnico relacionado con la seguridad o un riesgo de abuso en un producto de Google (SQLi, XSS, etc.)” y escribe “Go” como producto afectado.

Tracks

Dependiendo de la naturaleza de tu problema, el equipo de seguridad de Go lo clasificará como un problema en la categoría PÚBLICO, PRIVADO o URGENTE. A todos los problemas de seguridad se les emitirán números CVE.

PÚBLICO

Los problemas en la vía PÚBLICA afectan a configuraciones de nicho, tienen un impacto muy limitado o ya son ampliamente conocidos.

Los problemas de seguimiento PÚBLICO están etiquetados con Proposal-Security, discutido a través del proceso de revisión de la propuesta Go corregido en público y se obtiene una actualización para las próximas versiones menores ⬀ programadas. (que ocurren ~mensualmente). El anuncio del lanzamiento incluye detalles de estos problemas, pero no hay ningún anuncio previo.

Ejemplos de ediciones PÚBLICAS pasadas incluyen:

  • #44916 ⬀: archive/zip: puede entrar en pánico al llamar a Reader.Open
  • #44913 ⬀: encoding/xml: bucle infinito cuando se usa xml.NewTokenDecoder con un TokenReader personalizado
  • #43786 ⬀: cripto/elliptic: operaciones incorrectas en la curva P-224
  • #40928 ⬀: net/http/cgi,net/http/fcgi: Cross-Site Scripting (XSS) cuando el tipo de contenido no está especificado
  • #40618 ⬀: encoding/binary: ReadUvarint y ReadVarint pueden leer un número ilimitado de bytes de entradas no válidas
  • #36834 ⬀: crypto/x509: omisión de validación de certificados en Windows 10

PRIVADO

Los problemas en la vía PRIVADA son violaciones de propiedades de seguridad comprometidas.

Los problemas de las pistas PRIVADAS se solucionan en los próximos lanzamientos menores ⬀ programados y se mantienen privados hasta entonces.

De tres a siete días antes del lanzamiento, se envía un anuncio previo a golang-announce, anunciando la presencia de una o más correcciones de seguridad en las próximas versiones y si los problemas afectan a la biblioteca estándar, la cadena de herramientas o ambos, así como los ID de CVE reservados para cada una de las correcciones.

Algunos ejemplos de problemas PRIVADOS pasados incluyen:

  • #53416 ⬀: path/filePath: agotamiento de pila en Glob
  • #53616 ⬀: go/parser: agotamiento de pila en todas las funciones de Parse*
  • #54658 ⬀: net/http: maneja errores del servidor después de enviar GOAWAY
  • #56284 ⬀: syscall, os/exec: NUL no sanitizado en variables de entorno

URGENTE

Los problemas URGENTES en las pistas son una amenaza para la integridad del ecosistema de Go o están siendo explotados activamente, lo que provoca daños graves. No hay ejemplos recientes, pero incluirían la ejecución remota de código en net/http o la recuperación práctica de claves en crypto/tls.

Los problemas de seguimiento URGENTES se solucionan en privado y activan una versión de seguridad dedicada inmediata, posiblemente sin anuncio previo.

Marcar problemas existentes como relacionados con la seguridad

Si crees que un problema existente ⬀ está relacionado con la seguridad, te pedimos que envíes un correo electrónico a security@golang.org. El correo electrónico debe incluir el ID del problema y una breve descripción de por qué debe manejarse de acuerdo con esta política de seguridad.

Proceso de Divulgación

El proyecto Go utiliza el siguiente proceso de divulgación:

  1. Una vez que se recibe el informe de seguridad, se le asigna un controlador principal. Esta persona coordina el proceso de reparación y liberación.

  2. Se confirma el problema y se determina una lista del software afectado.

  3. El código se audita para encontrar posibles problemas similares.

  4. Si se determina, en consulta con el remitente, que se requiere un número CVE, el controlador principal obtendrá uno.

  5. Se preparan correcciones para las dos versiones principales más recientes y la revisión principal/master. Las correcciones se preparan para las dos versiones principales más recientes y se fusionan en head/master.

  6. En la fecha en que se aplican las correcciones, los anuncios se envían a golang-announce, golang-dev y golang-nuts.

Este proceso puede llevar algún tiempo, especialmente cuando se requiere coordinación con mantenedores de otros proyectos. Se hará todo lo posible para solucionar el error lo más oportunamente posible; sin embargo, es importante que sigamos el proceso descrito anteriormente para garantizar que las divulgaciones se manejen de manera consistente.

Para problemas de seguridad que incluyen la asignación de un número CVE, el problema se enumera públicamente en producto “Golang” en el sitio web CVEDetails así como en Sitio de divulgación nacional de vulnerabilidades.

Recibir actualizaciones de seguridad

La mejor manera de recibir anuncios de seguridad es suscribirte a la golang-announce. Cualquier mensaje relacionado con un problema de seguridad tendrá el prefijo [security].

Comentarios sobre esta política

Si tienes alguna sugerencia para mejorar esta política, presenta un issue ⬀ para discutirlo.
