Go Fuzzing

Go admite fuzzing en su cadena de herramientas estándar a partir de Go 1.18. Los tests nativos de Go Fuzz son impulsados por OSS-Fuzz.

Prueba el tutorial para hacer fuzzing con Go ⬀.

Descripción general

Fuzzing es un tipo de test automatizado que manipula continuamente las entradas de un programa para encontrar errores. Go Fuzzing utiliza la guía de cobertura para recorrer de manera inteligente el código que se está analizando para encontrar e informar fallas al usuario. Dado que puede llegar a casos extremos que los humanos a menudo pasan por alto, los tests fuzz pueden ser particularmente valiosas para encontrar vulnerabilidades y vulnerabilidades de seguridad.

A continuación se muestra un ejemplo de una prueba fuzz, destacando sus componentes principales.

Escribir tests fuzz

Requisitos

A continuación se detallan las reglas que deben seguir las pruebas fuzz.

  • Una prueba fuzz debe ser una función llamada FuzzXxx, que acepta solo un *testing.F y no tiene valor de retorno.
  • Las pruebas Fuzz deben estar en archivos *_test.go para ejecutarse.
  • Un objetivo fuzz debe ser una llamada a un método para (*testing.F).Fuzz que acepte un *testing.T como primer parámetro, seguido de los argumentos fuzzing. No hay valor de retorno.
  • Debe haber exactamente un objetivo de fuzz por prueba de fuzz.
  • Todas las entradas del seed corpus deben tener tipos que sean idénticos al argumentos fuzz, en el mismo orden. Esto es cierto para las llamadas a (*testing.F).Add y cualquier archivo de corpus en el directorio testdata/fuzz de la prueba fuzz.
  • Los argumentos fuzzing solo pueden ser de los siguientes tipos:
    • string, []byte
    • int, int8, int16, int32/rune, int64
    • uint, uint8/byte, uint16, uint32, uint64
    • float32, float64
    • bool

Sugerencias

A continuación encontrarás sugerencias que te ayudarán a aprovechar al máximo el fuzzing.

  • Los objetivos de fuzz deben ser rápidos y deterministas para que el motor de fuzz pueda funcionar de manera eficiente y se puedan reproducir fácilmente nuevas fallas y cobertura de código.
  • Dado que el objetivo (fuzz target) se invoca en paralelo entre varios trabajadores y en orden no determinista, el estado de un fuzz target no debería persistir más allá del final de cada llamada, y el comportamiento de un fuzz target no debería persistir más allá del final de cada llamada. Depende del estado global.

Ejecutar pruebas fuzz

Hay dos modos de ejecutar tu test fuzz: como test unitario (go test predeterminado) o con fuzzing (go test -fuzz=FuzzTestName).

Los tests Fuzz se ejecutan de manera muy similar a un test unitario de forma predeterminada. Cada entrada de la semilla corpus se probará con el fuzz target y se informará de cualquier error antes de salir.

Para habilitar el fuzzing, ejecuta go test con el indicador -fuzz, proporcionando una expresión regular que coincida con un solo test de fuzz. De forma predeterminada, todos los demás tests de ese paquete se ejecutarán antes de que comience el fuzzing. Esto es para garantizar que el fuzzing no informe ningún problema que ya habría sido detectado por un test existente.

Ten en cuenta que depende de ti decidir cuánto tiempo ejecutar el fuzzing. Es muy posible que una ejecución de fuzzing se ejecute indefinidamente si no encuentra ningún error. Habrá soporte para ejecutar estos tests fuzz continuamente usando herramientas como OSS-Fuzz en el futuro, consulta Issue n.º 50192.

Nota: Fuzzing debe ejecutarse en una plataforma que admita instrumentación de cobertura (actualmente AMD64 y ARM64) para que el corpus pueda crecer significativamente a medida que se ejecuta, y se puede cubrir más código mientras se realiza el fuzzing.

Salida de la línea de comando

Mientras el fuzzing está en progreso, el motor de fuzzing genera nuevas entradas y las ejecuta contra el objetivo de fuzzing proporcionado. De forma predeterminada, continúa ejecutándose hasta que se encuentra una entrada fallida o el usuario cancela el proceso (por ejemplo, con Ctrl^C).

El resultado se verá así:

~ go test -fuzz FuzzFoo
fuzz: elapsed: 0s, gathering baseline coverage: 0/192 completed
fuzz: elapsed: 0s, gathering baseline coverage: 192/192 completed, now fuzzing with 8 workers
fuzz: elapsed: 3s, execs: 325017 (108336/sec), new interesting: 11 (total: 202)
fuzz: elapsed: 6s, execs: 680218 (118402/sec), new interesting: 12 (total: 203)
fuzz: elapsed: 9s, execs: 1039901 (119895/sec), new interesting: 19 (total: 210)
fuzz: elapsed: 12s, execs: 1386684 (115594/sec), new interesting: 21 (total: 212)
PASS
ok      foo 12.692s

Las primeras líneas indican que la “cobertura de referencia” se recopila antes de que comience el fuzzing.

Para recopilar una cobertura de referencia, el motor de fuzzing ejecuta tanto la semilla de corpus como el corpus generado, para garantizar que no se produzcan errores y comprender la cobertura del código que el corpus existente ya proporciona.

Las siguientes líneas brindan información sobre la ejecución activa de fuzzing:

  • elapsed: la cantidad de tiempo que ha transcurrido desde que comenzó el proceso
  • execs: el número total de entradas que se han ejecutado contra el objetivo fuzz (con un promedio de ejecutivos/seg desde la última línea de registro)
  • new interesting: el número total de entradas “interesantes” que se han agregado al corpus generado durante esta ejecución de fuzzing (con el tamaño total de todo el corpus)

Para que una entrada sea "interesante", debe expandir la cobertura del código más allá de lo que puede alcanzar el corpus generado existente. Es típico que el número de nuevas entradas interesantes crezca rápidamente al principio y eventualmente disminuya, con explosiones ocasionales a medida que se descubren nuevas ramas.

Deberías esperar ver que el número "nuevo interesante" disminuya con el tiempo a medida que las entradas en el corpus comiencen a cubrir más líneas del código, con ráfagas ocasionales si el motor de fuzzing encuentra un nuevo path de código.

Entrada fallida

Puede ocurrir una falla durante el fuzzing por varias razones:

  • Se produjo un pánico en el código o en la test.
  • El fuzz target llamado t.Fail, ya sea directamente o mediante métodos como t.Error o t.Fatal.
  • Se produjo un error no recuperable, como un os.Exit o un desbordamiento de pila.
  • El fuzz target tardó demasiado en completarse. Actualmente, el tiempo de espera para la ejecución de un fuzz target es de 1 segundo. Esto puede fallar debido a un punto muerto o un bucle infinito, o debido al comportamiento previsto en el código. Ésta es una de las razones por las que se sugiere que el fuzz target sea rápido.

Si ocurre un error, el motor de fuzzing intentará minimizar la entrada al valor más pequeño posible y más legible por humanos, lo que aún producirá un error. Para configurar esto, consulta la sección configuración personalizada.

Una vez que se complete la minimización, se registrará el mensaje de error y el resultado terminará con algo como esto:

    Failing input written to testdata/fuzz/FuzzFoo/a878c3134fe0404d44eb1e662e5d8d4a24beb05c3d68354903670ff65513ff49
    To re-run:
    go test -run=FuzzFoo/a878c3134fe0404d44eb1e662e5d8d4a24beb05c3d68354903670ff65513ff49
FAIL
exit status 1
FAIL    foo 0.839s

El motor de fuzzing escribió esta entrada fallida en el corpus semilla para esa test de fuzz, y ahora será ejecutado de forma predeterminada con go test, que sirve como test de regresión una vez que se ha corregido el error.

El siguiente paso será diagnosticar el problema, corregir el error, verificar la solución volviendo a ejecutar go test y enviar el parche con el nuevo archivo testdata actuando como test de regresión.

Configuraciones personalizadas

La configuración predeterminada del comando go debería funcionar para la mayoría de los casos de uso de fuzzing. Por lo general, una ejecución de fuzzing en la línea de comando debería verse así:

$ go test -fuzz={FuzzTestName}

Sin embargo, el comando go proporciona algunas configuraciones cuando se ejecuta fuzzing. Estos están documentados en la documentación de cmd/go (paquete).

Por destacar algunos:

  • -fuzztime: el tiempo total o número de iteraciones que se ejecutará el fuzz target antes de salir, predeterminado de forma indefinida.
  • -fuzzminimizetime: el tiempo o número de iteraciones que se ejecutará el fuzz target durante cada intento de minimización, por defecto 60 segundos. Puede deshabilitar completamente la minimización configurando -fuzzminimizetime 0 al realizar el fuzzing.
  • -parallel: la cantidad de procesos de fuzzing que se ejecutan a la vez, predeterminado $GOMAXPROCS. Actualmente, configurar -cpu durante el fuzzing no tiene ningún efecto.

Formato del archivo Corpus

Los archivos Corpus están codificados en un formato especial. Este es el mismo formato tanto para el seed corpus como para el corpus generado.

A continuación se muestra un ejemplo de un archivo corpus:

go test fuzz v1
[]byte("hello\\xbd\\xb2=\\xbc ⌘")
int64(572293)

La primera línea se usa para informar al motor de fuzzing de la versión de codificación del archivo. Aunque actualmente no se planean versiones futuras del formato de codificación, el diseño debe admitir esta posibilidad.

Cada una de las líneas siguientes son los valores que componen la entrada del corpus y se pueden copiar directamente al código Go si lo deseas.

En el ejemplo anterior, tenemos un []byte seguido de un int64. Estos tipos deben coincidir exactamente con los argumentos fuzz, en ese orden. Un fuzz target para estos tipos se vería así:

f.Fuzz(func(*testing.T, []byte, int64) {})

La forma más fácil de especificar tus propios valores de semilla corpus es usar el método (*testing.F).Add. En el ejemplo anterior, se vería así:

f.Add([]byte("hello\\xbd\\xb2=\\xbc ⌘"), int64(572293))

Sin embargo, es posible que tengas archivos binarios grandes que preferirías no copiar como código en tu test y, en su lugar, permanecer como entradas de semilla corpus individuales en el directorio testdata/fuzz/{FuzzTestName}. La herramienta file2fuzz en golang.org/x/tools/cmd/file2fuzz se puede utilizar para convertir estos archivos binarios en archivos corpus codificados para []byte.

Para usar esta herramienta:

$ go install golang.org/x/tools/cmd/file2fuzz@latest
$ file2fuzz -h

Glosario

entrada de corpus: Una entrada en el corpus que se puede usar durante la fuzzing. Puede ser un archivo con formato especial o una llamada a (*testing.F).Add.

guía de cobertura: Un método de fuzzing que utiliza expansiones en la cobertura del código para determinar qué entradas del corpus vale la pena conservar para uso futuro.

entrada fallida: Una entrada fallida es una entrada del corpus que causará un error o entrar en pánico al correr contra el fuzz target.

fuzz target: La función del test fuzz que se ejecuta para las entradas del corpus y valores generados mientras se fuzzing. Se proporciona al test fuzz pasando la función a (*testing.F).Fuzz.

test fuzz: Una función en un archivo de test del formulario func FuzzXxx(*testing.F) que se puede utilizar para realizar fuzzing.

fuzzing: Un tipo de test automatizado que manipula continuamente las entradas de un programa para encontrar problemas como errores o vulnerabilidades a las que el código puede ser susceptible.

argumentos de fuzzing: Los tipos que se pasarán al objetivo de fuzzing, y mutado por el mutador.

motor de fuzzing: una herramienta que gestiona el fuzzing, incluido el mantenimiento del corpus, la invocación el mutador, identificando nuevas coberturas y reportando fallas.

corpus generado: Un corpus que es mantenido por el motor de fuzzing a lo largo del tiempo durante el fuzzing para realizar un seguimiento del progreso. Se almacena en $GOCACHE/fuzz. Estas entradas sólo se utilizan durante el fuzzing.

mutador: una herramienta utilizada durante el fuzzing que manipula aleatoriamente las entradas del corpus antes de pasarlas a un fuzz target.

paquete: Una colección de archivos fuente en el mismo directorio que se compilan juntos. Consulta la sección Paquetes ⬀ en la Especificación de lenguaje de Go.

seed corpus: Un corpus proporcionado por el usuario para un test de fuzz que puede ser utilizado para guiar el motor fuzzing. Se compone de las entradas del corpus proporcionadas por las llamadas f.Add dentro del test fuzz y los archivos en el directorio testdata/fuzz/{FuzzTestName} dentro del paquete. Estas entradas se ejecutan de forma predeterminada con go test, ya sea con fuzzing o no.

archivo de test: Un archivo del formato xxx_test.go que puede contener tests, puntos de referencia, ejemplos y tests fuzz.

vulnerabilidad: Una debilidad en el código sensible a la seguridad que puede ser explotada por un atacante.
